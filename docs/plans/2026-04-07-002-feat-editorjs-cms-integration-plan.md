# feat: Integrate Editor.js rich text editor with CMS

## Overview

Replace the plain `<Textarea>` body field in the CMS admin with a full-featured Editor.js block editor. Add custom block tools for mermaid diagrams (with live preview) and code blocks (with CodeMirror 6 syntax highlighting). Upgrade the web frontend `ContentBody` renderer to parse and display Editor.js block output, including server-rendered code highlighting via Shiki and client-rendered mermaid diagrams.

## Problem Frame

The CMS admin currently stores content body as plain text in a `<Textarea>`. Content creators need structured rich text editing — headings, lists, images, quotes, tables, code blocks with syntax highlighting, and embeddable mermaid diagrams. The web frontend renders body as `whitespace-pre-wrap` plain text, which was explicitly designed as "the future swap point for Editor.js" (see `apps/web/components/cms/content-body.tsx` and the CMS page rendering plan).

## Requirements Trace

- R1. Editor.js replaces the `<Textarea>` body field in content-form with a block-based rich text editor
- R2. Standard block plugins: Header, List, Image (with S3 upload), Quote, Table, Delimiter, Warning, Checklist, Embed (YouTube, Vimeo, etc.); inline tools: Inline Code, Marker, Underline; block tune: Text Variant Tune (call-out, citation, details); plugin: Undo/Redo
- R3. Custom mermaid diagram block tool with code textarea and debounced live SVG preview
- R4. Custom code block tool with CodeMirror 6 editor, language selector dropdown, and syntax highlighting
- R5. Serialization: Editor.js `OutputData` is `JSON.stringify()`'d into the existing `body: string | null` field
- R6. Deserialization: On edit, `JSON.parse(body)` loads existing blocks back into Editor.js
- R7. Web frontend `ContentBody` component renders Editor.js blocks as React components
- R8. Code blocks on web use Shiki for server-side syntax highlighting (zero client JS)
- R9. Mermaid blocks on web render client-side via `mermaid.render()`
- R10. Backward compatibility: existing plain-text body content still renders on the web frontend
- R11. Markdown import: when editing content with a legacy markdown/plain-text body, parse it into Editor.js blocks so the author can continue editing with the rich editor (headings, lists, code blocks, images, tables, quotes, mermaid fenced blocks all converted)

## Scope Boundaries

- **No backend API changes** — `body` remains `string | null`; Editor.js JSON is serialized as a string
- **No shared editor package** — editor components live in `apps/admin`, renderer components live in `apps/web`. Shared types are lightweight and can be co-located or extracted later
- **No drag-and-drop reordering plugin** — Editor.js has built-in block reordering via the toolbar; drag-and-drop can be added later via `editorjs-drag-drop`
- **No collaborative editing** — single-user editing only
- **No Attaches/Link preview plugins** — these require backend meta-fetching endpoints not yet available; can be added incrementally later
- **No alignment block tune** — `editorjs-alignment-blocktune` adds left/center/right alignment for any block. Useful for landing pages but adds complexity; deferred
- **No Raw HTML block** — security risk for user-generated content; deferred

## Key Technical Decisions

- **Use `@editorjs/editorjs` directly, no React wrapper library**: Community wrappers like `react-editor-js` are unmaintained (last published 4+ years ago) and incompatible with React 19. The imperative `useRef` + `useEffect` pattern is canonical for integrating DOM-manipulating libraries with React.

- **Lazy `import()` inside `useEffect` instead of `next/dynamic`**: Editor.js and all plugins require browser APIs. Dynamic `import()` inside `useEffect` is cleaner than nesting `next/dynamic` because it lets us import all plugin classes alongside the core in one initialization block, avoiding a chain of dynamic imports.

- **CodeMirror 6 for the admin code block, Shiki for web rendering**: CodeMirror 6 is modular (~300KB core), supports independent instances (critical for multiple code blocks), and has excellent editing capabilities. Shiki renders code in Server Components with zero client JS — ideal for the read-only web frontend. Monaco was rejected due to ~2.4MB+ bundle size and global state issues with multiple instances.

- **`JSON.stringify(outputData)` into the `body` field**: The API contract stores `body` as `string | null`. Storing serialized JSON avoids backend schema changes. The web renderer detects Editor.js JSON by parsing and checking for a `blocks` array, falling back to plain text for legacy content (R10).

- **Custom block tools as TypeScript classes**: Mermaid and Code blocks are custom Editor.js `BlockTool` implementations using the standard `render()`/`save()`/`destroy()` lifecycle. They render vanilla DOM (not React) inside Editor.js, which is the correct pattern since Editor.js manages its own DOM tree.

- **Mermaid live preview with 500ms debounce**: `mermaid.render()` is async and creates SVG. Debouncing prevents excessive renders during typing. Each render uses a unique ID to avoid collisions. Cleanup in `destroy()` clears timers and nulls DOM references.

- **`marked` for markdown-to-EditorJS conversion**: `marked.lexer()` (~12-14KB gzipped, zero dependencies) produces structured tokens that map almost 1:1 to Editor.js block types. A custom converter function (~150-200 lines) walks these tokens and emits `OutputData`. This avoids unmaintained community packages (`editorjs-md-parser`, etc.) which lack support for the List 2.0 format, mermaid blocks, tables, and inline formatting.

## Open Questions

### Resolved During Planning

- **How to integrate with `react-hook-form`?** Replace `register('body')` with a `Controller` wrapper. The Editor.js `onChange` callback calls `editor.save()` and sets the stringified JSON via `field.onChange(JSON.stringify(data))`. On form load, parse the body string back to `OutputData` for the editor's `data` prop.

- **Where do editor styles go?** Editor.js and custom block tool styles go in `apps/admin/app/globals.css`. The web renderer styles go in `apps/web` component-level Tailwind classes (mostly `prose` for typography).

- **How to handle image uploads in Editor.js?** Use `@editorjs/image` with a custom `uploader` config that calls the existing presign → S3 PUT → confirm flow. The uploader returns `{ success: 1, file: { url: publicUrl } }` matching the plugin's expected response format.

- **Language switching in CodeMirror 6?** Use CodeMirror's Compartment API to dynamically swap language extensions when the user changes the language dropdown. Pre-bundle common languages (JavaScript, TypeScript, Python, HTML, CSS, JSON, Go, SQL) and lazy-load others.

- **How to handle legacy markdown body on edit?** Use `marked.lexer()` to parse the markdown into tokens, then convert each token to an Editor.js block. The converter is a pure function (`markdownToEditorJS(md: string): OutputData`) that runs during deserialization in `content-editor.tsx`. Detection: if `body` is a non-empty string that fails `JSON.parse()` (or parses but lacks a `blocks` array), treat it as markdown and run the converter. The original markdown is not lost — it is transformed into Editor.js blocks on first edit, and re-saved as Editor.js JSON on form submit.

### Deferred to Implementation

- **Exact set of CodeMirror language extensions to bundle**: Start with 8-10 common languages, expand based on user feedback
- **Block alignment tune**: Whether to add `editorjs-alignment-blocktune` for left/center/right alignment — start without and add if needed for landing page use cases
- **Mermaid theme synchronization**: Whether mermaid diagram theme should match the admin's light/dark mode — start with `default` theme

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

```mermaid
graph TB
    subgraph Admin ["apps/admin (Editor)"]
        CF[ContentForm] -->|Controller| EW[ContentEditor]
        EW -->|deserialize body| DES{JSON.parse?}
        DES -->|valid EditorJS JSON| EJS[Editor.js Core]
        DES -->|not JSON / no blocks| MDC[markdownToEditorJS<br/>marked.lexer → blocks]
        MDC --> EJS
        EJS --> STD[Standard Plugins<br/>Header, List, Image, Quote,<br/>Table, Delimiter, Warning,<br/>Checklist, Embed]
        EJS --> INL[Inline Tools<br/>Marker, InlineCode, Underline]
        EJS --> TUN[Tunes & Plugins<br/>TextVariantTune, Undo/Redo]
        EJS --> MBT[MermaidTool<br/>textarea + debounced SVG preview]
        EJS --> CBT[CodeBlockTool<br/>CodeMirror 6 + language selector]
        STD -->|Image uploader| S3[Presign → S3 PUT → Confirm]
        EJS -->|onChange → save()| SER[JSON.stringify → body field]
    end

    subgraph Web ["apps/web (Renderer)"]
        CB[ContentBody] -->|JSON.parse| ER[EditorRenderer]
        ER --> SB[Static Blocks<br/>Header, Paragraph, List,<br/>Quote, Table, Delimiter,<br/>Warning, Checklist, Image<br/><i>Server Components</i>]
        ER --> ShB[CodeBlock<br/>Shiki server-side highlighting<br/><i>async Server Component</i>]
        ER --> MB[MermaidBlock<br/>mermaid.render() client-side<br/><i>'use client'</i>]
        CB -->|plain text fallback| PT[Legacy plain text render]
    end

    SER -->|API: body string| DB[(Backend API<br/>body: string)]
    DB -->|body string| CB
```

**Data flow:**
1. **Write path**: User edits in Editor.js → `onChange` calls `editor.save()` → `OutputData` is `JSON.stringify()`'d → stored in react-hook-form's `body` field → submitted to API as a string
2. **Read path (admin — Editor.js JSON)**: API returns `body` string → `JSON.parse()` succeeds with `blocks` array → passed as `data` to Editor.js constructor → blocks render in editor
3. **Read path (admin — legacy markdown)**: API returns `body` string → `JSON.parse()` fails or no `blocks` → `markdownToEditorJS()` converts markdown tokens to `OutputData` → blocks render in editor. On next save, content is stored as Editor.js JSON (one-time migration)
4. **Read path (web)**: API returns `body` string → `ContentBody` tries `JSON.parse()` → if valid Editor.js JSON with `blocks`, renders via `EditorRenderer` mapping each block type to a React component → falls back to plain text for legacy content

- [ ] **Unit 3: Custom mermaid diagram block tool**

  **Goal:** Create a custom Editor.js block tool that provides a code textarea for mermaid syntax with a debounced live SVG preview.

  **Requirements:** R3

  **Dependencies:** Unit 1

  **Files:**
  - Create: `apps/admin/components/editor/tools/mermaid-tool.ts`
  - Modify: `apps/admin/components/editor/plugins.ts` (register MermaidTool)

  **Approach:**
  - Implement as a TypeScript class conforming to the Editor.js `BlockTool` interface
  - `static get toolbox()` returns a mermaid-themed icon and "Mermaid Diagram" title
  - `static get isReadOnlySupported()` returns `true`
  - `render()` creates a wrapper `<div>` with two children: a `<textarea>` for mermaid code input and a `<div>` for the SVG preview. In read-only mode, only the preview div is rendered
  - Text input has an `input` event listener that debounces (500ms) before calling `_renderMermaid(code)`
  - `_renderMermaid()` calls `mermaid.render(uniqueId, code)` asynchronously, injects the returned SVG into the preview div. On error, shows the error message in a styled `<pre>` element
  - `save()` returns `{ code: string }` from the textarea value
  - `validate(data)` checks that `code` is a non-empty string
  - `destroy()` clears the debounce timer, removes event listeners, nulls DOM references

  - Use `crypto.randomUUID()` or a counter prefix for mermaid render IDs to avoid collisions when multiple mermaid blocks exist on the same page
  - `mermaid.initialize({ startOnLoad: false, theme: 'default' })` is called once at module level
  - CSS for the split-pane layout (textarea left/top, preview right/bottom) added to `apps/admin/app/globals.css`

  **Patterns to follow:**
  - Editor.js BlockTool API: constructor receives `{ data, api, config, readOnly }`
  - Mermaid render pattern: unique ID per render call, async/await, error catch

  **Test scenarios:**
  - Happy path: User types `graph TD; A-->B;` → after 500ms debounce, SVG diagram appears in preview
  - Happy path: Saving the block returns `{ code: "graph TD; A-->B;" }`
  - Happy path: Loading existing data shows pre-populated textarea and rendered diagram
  - Edge case: Invalid mermaid syntax → error message shown in preview, not a crash
  - Edge case: Empty code → placeholder message shown, validate returns false (block skipped on save)
  - Edge case: Rapid typing → only the final debounced value triggers a render
  - Edge case: Read-only mode → no textarea, only the rendered SVG diagram

  **Verification:**
  - Mermaid block appears in the toolbox, renders diagrams live, and saves/loads code correctly

  - Use `marked.lexer(markdown)` to tokenize the markdown into structured tokens without producing HTML
  - Walk the token array and convert each token to an Editor.js block:
    - `heading` → `{ type: 'header', data: { text, level } }` — inline formatting rendered to HTML via `marked.parseInline()`
    - `paragraph` → `{ type: 'paragraph', data: { text } }` — inline tokens (bold, italic, links, code) rendered to HTML. If the paragraph contains only a single `image` token, promote it to an Image block instead
    - `code` (fenced) → `{ type: 'code', data: { code, language } }` — **except** when `lang === 'mermaid'`, which produces `{ type: 'mermaid', data: { code } }`
    - `blockquote` → `{ type: 'quote', data: { text, caption } }` — first inner paragraph as `text`, subsequent as `caption`
    - `list` → `{ type: 'list', data: { style, items } }` — recursive conversion matching the List 2.0 `{ content, meta, items }` format. Task list items (`- [x]`, `- [ ]`) produce checklist style with `meta: { checked }`
    - `table` → `{ type: 'table', data: { withHeadings: true, content: string[][] } }` — header row + data rows flattened to 2D array
    - `hr` → `{ type: 'delimiter', data: {} }`
    - `space` → skip
    - Unknown/unsupported tokens → skip with console warning
  - Use `marked.parseInline(text)` to render inline markdown formatting to HTML for text fields (bold → `<b>`, italic → `<i>`, code → `<code>`, links → `<a>`, etc.)
  - Generate unique block IDs using a counter or `crypto.randomUUID()`

  **Patterns to follow:**
  - Pure utility function pattern like `apps/admin/lib/slugify.ts`
  - `marked.lexer()` API for token access without HTML rendering

  **Test scenarios:**
  - Happy path: `# Hello\n\nWorld` → Header block (level 1, text "Hello") + Paragraph block (text "World")
  - Happy path: `**bold** and *italic*` → Paragraph with `<b>bold</b> and <i>italic</i>`
  - Happy path: Fenced code block with language → Code block with `{ code, language }`
  - Happy path: Fenced code block with `mermaid` language → Mermaid block `{ code }`
  - Happy path: `![alt](url "caption")` as sole paragraph content → Image block `{ file: { url }, caption }`
  - Happy path: Unordered list with nested sub-list → List block with recursive `items` array
  - Happy path: GFM table with header → Table block with `withHeadings: true` and 2D `content` array
  - Happy path: `> Quote text` → Quote block `{ text, caption: '' }`
  - Happy path: `---` → Delimiter block
  - Happy path: `- [x] Done\n- [ ] Todo` → List block with `checklist` style and `meta.checked` values
  - Edge case: Empty string → OutputData with empty `blocks` array
  - Edge case: Mixed content (heading + paragraph + code + list + image + table) → all block types converted in correct order
  - Edge case: Multi-paragraph blockquote → first paragraph as `text`, rest joined as `caption`
  - Edge case: Paragraph with inline image mixed with text → stays as paragraph (image not promoted)
  - Edge case: Code block with no language → Code block with `language: 'plaintext'`

## System-Wide Impact

- **Interaction graph:** The change touches two apps — `apps/admin` (editor) and `apps/web` (renderer). No middleware, auth, or API route changes. The Image plugin uploader calls existing proxy routes (`/api/cms/uploads/presign`, `/api/cms/uploads/confirm`) — no new API surface.
- **Error propagation:** Editor.js initialization failure in admin is contained to the body field — the rest of the form works. Mermaid/code render failures on the web are caught per-block and don't break the page. `JSON.parse` failure on the web triggers the plain text fallback.
- **State lifecycle risks:** Editor.js manages its own DOM tree outside React. The `useEditorJS` hook must strictly guard against double-initialization (React 19 StrictMode) and call `destroy()` on unmount. CodeMirror instances inside code blocks must also be destroyed. Mermaid debounce timers must be cleared. Tab switching in the form must use `forceMount` to prevent editor destruction.
- **Component boundary shift:** `ContentBody` on the web side changes from a simple synchronous function to a component that orchestrates both Server Components (most blocks, async Shiki code blocks) and a Client Component (mermaid block via `next/dynamic`). Both parent pages (`[...slug]/page.tsx` and `blog/[slug]/page.tsx`) are already async Server Components, so this is compatible.
- **API surface parity:** The `body` field contract (`string | null`) is unchanged. No other interfaces are affected.
- **Unchanged invariants:** The content form's other tabs (Taxonomy, SEO, Settings), the content list page, upload management, author/category/tag CRUD — none are modified. The API types, Zod schema shape (body remains `z.string().nullable().optional()`), and form submission logic are preserved.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| Editor.js bundle size adds to admin app load time | Lazy import inside `useEffect` ensures it loads only when the form mounts, not on initial page load. Loading skeleton provides visual feedback |
| `dangerouslySetInnerHTML` for block rendering on web (XSS risk) | Editor.js sanitizes inline HTML during save via per-tool `sanitize` config. Content is authored by admin users (not public). Server-side sanitization can be added later if needed |
| Mermaid `mermaid.render()` can throw on invalid syntax | Wrapped in try/catch in both the admin live preview and web renderer. Error state shows the error message, not a blank block |
| CodeMirror 6 language packages increase admin bundle | Only common languages are bundled. Language packages are individually ~5-15KB gzipped. Total is ~100-150KB for 8-10 languages |
| Editor.js plugin compatibility with v2.31.x | All block/inline/tune plugins are official `@editorjs/` packages. The sole community plugin (`editorjs-undo`) is widely used and essential — Editor.js has no built-in undo. If it breaks, undo can be removed without affecting content serialization |
| Legacy plain text content after migration | Explicit backward compatibility: `ContentBody` detects JSON vs plain text. No forced migration of existing content |
| Keyboard/paste event conflicts between Editor.js and CodeMirror/mermaid | Custom block tools use `event.stopPropagation()` on their containers for `keydown` and `paste` events to prevent Editor.js from hijacking input |
| Tab switching destroys Editor.js instance | `forceMount` on Main tab's `TabsContent` keeps the editor alive across tab switches. Hidden with CSS `data-[state=inactive]:hidden` |
| `editor.save()` called on every keystroke is expensive for large documents | 300ms debounce in the `useEditorJS` hook's onChange pipeline prevents per-keystroke serialization |
| Client-side HTML sanitization is a UX guardrail, not a security boundary | Acceptable for admin-only authoring. Server-side sanitization can be added if the authoring surface expands beyond trusted admin users |
